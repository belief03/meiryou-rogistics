## 7. 追跡機能と先方システム連携（参考）

### 連携は可能か

**可能です。** 先方（運送会社）が次のいずれかを持っていれば、HP の「配達状況の確認」から連携できます。

| パターン | 内容 |
|----------|------|
| **自社の追跡システム** | 基幹・配送システムに伝票番号検索や Web 画面がある場合 |
| **外部サービス** | 伝票番号検索 API や追跡専用 SaaS を利用している場合 |
| **なし** | 当面は「お問い合わせ」や電話案内のみで運用する場合 |

### 連携の流れ（イメージ）

1. **ヒアリング** … 先方の「今の追跡の仕組み」を確認（自社システム／外部サービス／なし）。
2. **方式の決定** … 以下のどれで実装するか検討。
   - **API 連携** … 先方システム or 外部サービスの API を呼び出し、HP 上に結果を表示。
   - **リンク・リダイレクト** … 「配達状況の確認」ボタンで先方の追跡ページへ遷移（別タブなど）。
   - **iframe 埋め込み** … 先方の追跡画面を HP 内に表示（許可・技術的制約の確認が必要）。
3. **仕様・環境の確認** … API の有無、認証方法、レスポンス形式、利用規約・SLA。
4. **開発・テスト** … 伝票番号入力 → 問い合わせ → 結果表示（または遷移）まで動作確認。
5. **公開・運用** … 本番環境で公開し、障害時や仕様変更時の連絡体制を決めておく。

### どれを選ぶか（API が常にベストとは限らない）

| 方式 | 向いている場合 | メリット | デメリット |
|------|----------------|----------|------------|
| **API 連携** | 先方 or 外部に API があり、HP 上でデザインを統一したいとき | 見た目・体験を自社 HP に統一できる。遷移がなくスムーズ。 | 開発・テスト・保守の工数が増える。API 仕様変更の影響を受ける。 |
| **リンク・リダイレクト** | すでに追跡ページ（自社 or 外部）があり、とにかく早く提供したいとき | 実装が簡単でコストが低い。先方の画面更新に追随しやすい。 | HP から離れるため、デザインの一体感は薄れる。 |
| **iframe 埋め込み** | 追跡画面はそのまま使いたいが、HP 内に収めたいとき | 見た目は「HP 内」にできる。API がなくても可能な場合がある。 | クリックジャッキング対策で iframe 禁止のサイトも多い。表示崩れ・レスポンシブの調整が難しい。 |

**結論** … 「HP 上で見た目まで統一したい」「API が用意されている」なら **API 連携がベスト**。  
「まずは追跡できる入口を安く・早く用意したい」「API がない or 不明」なら **リンク・リダイレクト** から始めて、必要になったら API 連携に切り替えるのが現実的です。

**安全面・使いやすさを最優先する場合**  
→ **API 連携** が有利です。HP 内で完結するため「どこに飛ばされるか」の不安がなく、表示も自社で制御できるので、エラーメッセージやプライバシー配慮の文言を統一しやすく、信頼感のある体験にしやすいです。

### 追跡機能がない場合に「作る」ことは可能か

**可能です。** 次のような形で検討できます。

| パターン | 内容 | 想定工数・条件 |
|----------|------|----------------|
| **追跡「画面」だけ先に用意** | 伝票番号入力フォームと結果表示のUIをHPに実装。中身は「準備中」や「お問い合わせください」にしておき、後からAPIや自社システムと接続。 | 少なめ。のちの連携を見据えた設計にしておく。 |
| **外部の追跡API・SaaS を利用** | 伝票番号で検索できる外部サービス（業界共通や運送向けのAPI）を調べ、HPからそのAPIを呼んで結果を表示。先方の基幹システムがなくても「追跡できるHP」にできる。 | 中。利用規約・費用・伝票体系の一致要確認。 |
| **自社でステータスを登録する仕組みから作る** | 管理画面で伝票番号と配達状況を登録し、HPの検索ではそのデータを参照する。つまり「追跡システム」を小規模に新規構築。 | 大。バックエンド・DB・管理画面・運用ルールが必要。HP制作の範囲を超える場合あり。 |

打ち合わせでは「今は追跡の仕組みがないが、HPで検索できるようにしたい」かどうかを聞き、**ない場合**は「まずは入口（画面）だけ用意して後から連携」か「外部APIの利用を検討」か「自社で登録する仕組みまで作るか」を、予算・スケジュール・運用と合わせて提案するとよいです。

### 注意点・確認したいこと

| 観点 | 確認内容 |
|------|----------|
| **API の有無** | 自社システムに伝票検索用 API があるか。外部サービスなら API 提供の有無と仕様書。 |
| **認証・鍵** | API キーやトークンが必要か。HP から呼ぶ場合、鍵の管理方法（環境変数など）をどうするか。 |
| **データ形式** | 伝票番号の形式（桁数・チェック等）。返却データの項目（ステータス・日時・拠点名など）。 |
| **利用制限** | リクエスト数制限、利用可能時間帯、利用規約上の制約。 |
| **障害時** | 先方システムや外部 API が落ちたときの表示（「只今確認できません」等）と案内文。 |
| **セキュリティ** | 伝票番号や配達情報を扱うため、HTTPS 必須。個人情報扱いの有無も確認。 |
| **保守** | 先方システムの仕様変更・API バージョンアップ時の対応窓口と費用感。 |

打ち合わせでは「追跡は自社システム／外部サービス／なしのどれを想定か」を聞き、**連携する場合**は上記を踏まえて「API やリンク先の有無」「障害時の案内」まで一緒に決められるとスムーズです。

## 8. デザイン方針（CSS-B：Keyence寄せ）

### ねらい

- **「名刺代わりの看板」**として、最初の1画面〜2画面で「何の会社か」「何を大切にしているか」「誰向けか」「強み」を伝える。
- 機能訴求よりも、**信頼感・事業の輪郭・法人向けの安心感**を重視。

### 追加したCSSブロック（2つ）

| ブロック | 目的 | 伝える内容 |
|----------|------|------------|
| **Brand Message** | 1スクロール目で業務領域を明確化 | 何をしている会社か |
| **Brand Highlights（3カード）** | 一目で判断できる情報圧縮 | 価値観・ターゲット・強み |

### 壁打ち（より良くするための確認）

| 観点 | 壁打ちの質問 |
|------|--------------|
| **看板としての一言** | 「当社は○○の会社です」を15〜25文字で言い切るなら、どの文が最適か？ |
| **価値観** | 「安全・確実・誠実」以外で、代表の言葉として最優先したい価値は何か？ |
| **ターゲット** | 最優先の顧客は「設備メーカー」「工場」「商社」のどれか？優先順をつけると？ |
| **強みの根拠** | 強みを裏づける具体事例（件数、年数、対応エリア、実績企業）は何を公開できるか？ |
| **印象設計** | 「堅実」「先進」「親しみ」のうち、サイト全体で最も強く出したい印象はどれか？ |
| **次アクション** | 閲覧者に最初に取ってほしい行動は「問い合わせ」「会社情報閲覧」「サービス確認」のどれか？ |
